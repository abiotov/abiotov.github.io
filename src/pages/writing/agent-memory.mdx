Every agent framework now ships a "memory" feature. Every vendor claims state-of-the-art
recall. Every demo shows an assistant that remembers your dog's name. And yet: the same
system's published benchmark score moves by **25 points** depending on who runs the
evaluation, and the most-cited memory benchmark turns out to have a broken answer key
and a judge that accepts most wrong answers.

I spent the last week going deep on this. I read the recent surveys, traced the papers
they cite back to primary sources, and fact-checked the headline numbers one by one;
every figure in this post survived that check unless I flag it otherwise. What came out
is the map I wish someone had handed me at the start: what exists, what is actually
broken, and which directions have real evidence behind them.

## Why memory is the lever that moves everything

Start with where an agent's money goes. A single-turn LLM call is priced roughly by the
tokens it generates. An agent is a different animal, because it loops, and a reasonable
first-order model of its cost (adapted from [Yang et al., 2026](https://arxiv.org/abs/2601.14192))
looks like this:

$$
\mathrm{Cost}_{\text{agent}} \;\approx\; \alpha N_{\text{tok}} \;+\; I_{\text{tool}}\cdot C_{\text{tool}} \;+\; I_{\text{mem}}\cdot C_{\text{mem}} \;+\; I_{\text{retry}}\cdot C_{\text{retry}}
$$

where the indicator terms fire on every tool invocation, memory operation, and retry
along the trajectory. The structural problem is the loop itself: everything the agent
observes at turn $n$ becomes input it pays for at turn $n{+}1$, so context grows
superlinearly with task length.

<FigLoop num="1" />

Look at where memory sits in that loop. It is the only component that can shrink every
cost term at once: remembered constraints mean fewer retries, cached tool outcomes mean
fewer redundant calls, recalled plans mean shallower deliberation, and a compact
retrieved context means fewer tokens per step. That is why I think memory is the most
underrated engineering surface in agents right now: it is not a feature, it is the cost
model.

The organizing idea I'll use for the rest of this post comes from the
[*Toward Efficient Agents* survey](https://arxiv.org/abs/2601.14192): treat memory as a
**lifecycle** with three stages: construction (how raw experience becomes stored
memory), management (how the store stays healthy over time), and access (how the right
memories come back into context).

<FigLifecycle num="2" />

The trap the lifecycle view exposes: optimizing one stage just moves the bill. Compress
aggressively at write time and you pay in repair later, because the detail you need is
gone. Skip maintenance and retrieval drowns in stale duplicates. Crank up retrieval
recall and you rebuild the exact long-context cost that memory was supposed to
eliminate.

## A map of what exists

### Construction: three substrates

The deepest split in the field is *what physical form* the memory takes.

**Textual memory** is the practical mainstream: memory as readable text or structured
records. Within it, four sub-families in increasing order of structure:

| Family | Idea | Representative systems |
|---|---|---|
| Prompt-resident | one compact state, rewritten every turn inside the context | [MemAgent](https://arxiv.org/abs/2507.02259), [MEM1](https://arxiv.org/abs/2506.15841) |
| Item-based | discrete facts/insights in an external store, with CRUD | [Mem0](https://arxiv.org/abs/2504.19413), [A-MEM](https://arxiv.org/abs/2502.12110), [ReasoningBank](https://arxiv.org/abs/2509.25140), ACE |
| Graph-based | entities and relations as nodes/edges, time-aware | [Zep](https://arxiv.org/abs/2501.13956), AriGraph, MAGMA |
| Hierarchical | explicit tiers with promotion/eviction, OS-style | [MemGPT](https://arxiv.org/abs/2310.08560), MemoryOS, LightMem |

A few of these deserve a sentence of color. [MemGPT](https://arxiv.org/abs/2310.08560)
is the founding metaphor: treat the context window as RAM, external storage as disk, and
let the model page between them with tool calls. [A-MEM](https://arxiv.org/abs/2502.12110)
runs a Zettelkasten: every experience becomes an atomic note that autonomously links to
and even rewrites related older notes. [Zep](https://arxiv.org/abs/2501.13956) builds a
temporal knowledge graph where each fact edge carries a validity interval, so "the user
moved to Lyon" can supersede "the user lives in Paris" instead of contradicting it.
ReasoningBank distills strategies from *failed* trajectories as well as successful ones,
which sounds obvious and is done almost nowhere else.

**Latent memory** skips text entirely and stores experience as neural activations:
compressed KV-cache states ([MemoRAG](https://arxiv.org/abs/2409.05591), Activation
Beacon), or latent sequences injected on demand into the reasoning stream (MemGen).

**Parametric memory** goes furthest and writes experience into weights.
[MemoryLLM](https://arxiv.org/abs/2402.04624) bolts a self-updating pool of memory
tokens onto a frozen backbone; [Titans](https://arxiv.org/abs/2501.00663) trains a
neural memory module *at test time*, writing hard only when prediction error is high,
a mechanism its authors explicitly frame as surprise.

<FigTradeoff num="3" />

The trade-off in Fig. 3 is, in my view, the quiet reason the market looks the way it
does. Latent and parametric memory can be dramatically cheaper per query. But the stored
knowledge becomes uninspectable: you cannot audit it, edit a wrong fact, cite a source,
or migrate it when you switch models. Text is expensive and slow, but it is *governable*.
Every production system I can find sits at the textual end, and I don't think that's an
accident.

### Management: keeping the store from rotting

Once memories accumulate, someone has to garbage-collect. Three regimes exist:

- **Rule-based**: recency decay, FIFO eviction, capacity caps. Cheap, predictable, and
  blind: the rules are proxies for usefulness, and (see below) they are bad proxies.
- **LLM-based**: the model decides what to add, merge, rewrite, delete. Semantically
  aware, but every maintenance decision now costs a model call, and one bad merge
  corrupts the store silently. There is no error message for a wrong memory.
- **Hybrid**: cheap triggers handle routine operations; the LLM is reserved for
  consolidation and conflict resolution. This is where the production-minded designs
  converge (MemoryOS's tiered triggers, LightMem's offline consolidation, Zep's
  conflict-checking on graph updates).

### Access: where memory helps or actively hurts

Retrieval failure is asymmetric in a way that took me a while to appreciate. Miss one
critical memory and the agent acts wrongly or re-explores what it already learned.
Retrieve too generously and you re-inflate the context you were trying to shrink, and
bury the model in near-duplicates.

The interesting work here rejects the default assumption that vector similarity is the
right selector. **Similarity is not utility**: as a store grows, embedding search
increasingly returns things that *look like* the query without *helping* the task.
Three papers attack this directly by learning retrieval from outcomes: RMM trains a
reranker online with RL, [Memento](https://arxiv.org/abs/2508.16153) learns a Q-function
over past cases, and MemRL retrieves by similarity first and then filters by learned
value. Different mechanisms, same conclusion: let task success, not cosine distance,
decide what gets remembered *into the context*.

Two extensions complete the map. **Skills** are procedural memory: instead of recording
what happened, store verified procedures for how to act, and amortize their construction
cost across recurring tasks. And **multi-agent memory** re-runs the whole lifecycle at
team scale, where the unsolved parts are provenance (who wrote this, from what evidence?)
and access control on shared stores.

## The benchmark reality check

This is the section I most wanted someone else to have written, because the numbers
here shape purchasing decisions and none of them mean what they appear to mean.

<FigScores num="4" />

Fig. 4 is not a cherry-picked anecdote; it is the current state of evaluation in this
field. The same system, Zep, has four published LoCoMo scores spanning 25.6 points:
[Mem0's comparison paper](https://arxiv.org/abs/2504.19413) measured it at 65.99, Zep
[responded](https://blog.getzep.com/lies-damn-lies-statistics-is-mem0-really-sota-in-agent-memory/)
that its configuration was broken and published a corrected 75.14 (after retracting its
own earlier 84), and Mem0 [re-audited](https://github.com/getzep/zep-papers/issues/5)
that claim across ten seeds and got 58.44. Every number is vendor-produced. There is no
neutral referee.

It gets worse, because the battlefield itself is compromised:

- **LoCoMo's answer key is broken.** A systematic audit by Penfield Labs
  ([write-up](https://dev.to/penfieldlabs/we-audited-locomo-64-of-the-answer-key-is-wrong-and-the-judge-accepts-up-to-63-of-intentionally-33lg),
  [audit repo](https://github.com/dial481/locomo-audit)) found 99 score-corrupting
  errors in the 1,540-question ground truth (6.4%): hallucinated facts, wrong dates,
  answers attributed to the wrong speaker.
- **The judge is worse than the key.** The same audit planted intentionally wrong
  answers and found the commonly used gpt-4o-mini judge accepted **62.81%** of
  vague-but-on-topic wrong answers, and 10.61% of specifically-wrong ones (wrong name,
  wrong date). Any score difference smaller than the judge's false-acceptance rate is
  noise wearing a lab coat.
- **Marketing pages drift from papers.** Mem0's research page advertised 92.5 on LoCoMo
  as of July 2026; its own peer-reviewed paper reports 67.13 (68.44 for the graph
  variant), and a [GitHub issue](https://github.com/mem0ai/mem0/issues/3944) documents a
  failed independent reproduction of the marketing figure. No judge model is disclosed
  for the page.
- **The serious benchmark tells a harder story.**
  [LongMemEval](https://arxiv.org/abs/2410.10813) (ICLR 2025) tests five abilities:
  information extraction, multi-session reasoning, temporal reasoning, knowledge
  updates, and abstention. Long-context models drop 30 to 60% accuracy versus an oracle
  on it, and the two abilities where everything fails hardest are **temporal ordering**
  and **knowledge updates**: exactly the two things a "memory" is supposedly for.
- **Scale quietly degrades everything.** On the third-party
  [BEAM benchmark](https://arxiv.org/abs/2510.27246) (ICLR 2026), Mem0 self-reports
  64.1 at 1M tokens of history and 48.6 at 10M. Different question sets, so read it
  directionally, but the vendor's own framing concedes that at 10M the system "cannot
  always surface the exact correct memory".
- **Nobody measures the lifecycle.** The entire premise of the category is that memory
  *improves with use*. As of early 2026, no benchmark measured whether a store gets
  better or drifts worse over its lifetime. The first attempts are only now appearing
  ([DriftBench](https://arxiv.org/abs/2605.10990), [Library Drift](https://arxiv.org/abs/2605.19576)).

<Callout title="How to read any memory benchmark claim">
Four questions. Which judge model, with what prompt, validated how? Who configured the
competing systems, and did those vendors review the configurations? Has the answer key
been audited? Are cost, latency and inserted tokens reported next to accuracy? I have
not found a single published comparison, academic or commercial, that clears all four.
</Callout>

## Twelve documented ways current systems break

Everything below is grounded in either the surveys or a verified primary source. I've
ordered them roughly by how underappreciated I think they are.

**1. Errors are cumulative and persistent.** This is *the* structural difference from
RAG. A retrieval error in RAG is confined to one query; in write-enabled memory, one
wrong write contaminates every future read that touches it. The
[SSGM paper](https://arxiv.org/abs/2603.11768) makes the argument formally, error
propagation has been [measured empirically](https://arxiv.org/abs/2505.16067), and one
paper found corrupted memories that persist and self-reinforce, earning the best title
of the year: [Zombie Agents](https://arxiv.org/abs/2602.15654).

**2. Semantic drift by re-encoding.** Raw text becomes a summary, the summary becomes a
reflection, and every hop strips nuance until the stored "fact" no longer matches what
happened; [measured](https://arxiv.org/abs/2512.13564) in MemGPT/Mem0-style pipelines.
The mechanism should feel familiar: it is the telephone game, run by a language model,
against your own history.

**3. Memory poisoning is an open attack surface.** Content injected into memory persists
and reactivates across sessions: [attacks are documented](https://arxiv.org/abs/2601.05504),
[defenses are early](https://arxiv.org/abs/2605.23723), and no deployed system has a
convincing answer. A memory without provenance is an unlocked door that stays unlocked.

**4. Naive forgetting destroys performance.** The cleanest evidence is A-MEM's own
ablation: Ebbinghaus-style time decay controls store size and latency, and substantially
degrades task success. Time-since-last-access is simply a bad proxy for value; the fact
you need next week may not have been touched in months.

**5. Over-compression loses decision-critical detail.** LightMem measured the curve:
accuracy falls as compression rises. The objective should be task-aware (preserve causal
facts, constraints, failures, procedures), not "make it shorter".

**6. LLM-based maintenance is expensive and fragile.** Per-turn synchronous updates
multiply calls and latency; asynchronous consolidation is cheaper but adapts slower; and
either way a wrong merge fails silently.

**7. Similarity is not utility** (see above), and it gets *worse* as the store grows,
which is the opposite of what a memory should do.

**8. Costs escalate with lifetime.** Without deduplication, selective retention and
pruning, storage and indexing overhead grows until it eats the efficiency gains that
justified the memory in the first place.

**9. Temporal reasoning and knowledge updates are the weakest abilities**, per
LongMemEval, and most stores have no first-class representation of time: no validity
intervals, no superseding relations, no event order. Zep and MAGMA are structural
exceptions; demonstrated results are still thin.

**10. Latent and parametric memory trade away auditability**, and may simply relocate
drift into catastrophic forgetting inside the weights. Open question; nobody has solid
numbers for this family yet.

**11. Multi-agent shared memory lacks provenance and access control.** Concurrent
writes, stale shared state, and no story for who is allowed to believe what.

**12. Evaluation itself is broken**, per the previous section, which means a system can
be state-of-the-art on paper and worse than a naive baseline in your product. You
cannot currently know without measuring it yourself.

## Directions with actual evidence behind them

Filtering aggressively for ideas with verified numbers or peer review, rather than
launch-post energy:

**Sleep-time compute.** [Lin et al.](https://arxiv.org/abs/2504.13171) (Letta and UC
Berkeley) let the system think about its context offline, before queries arrive:
pre-digesting raw history into learned context during idle time. Reported: about **5x
less test-time compute** at equal accuracy, up to +13/+18% accuracy from scaling the
offline budget, and 2.5x lower average cost when the offline work amortizes across
related queries. Honest caveats: the authors built the benchmarks, Letta's founders are
co-authors, and I found no independent replication yet. But it attacks the
maintenance-cost problem (limit 6) at its root, and biology quietly agrees with the
design.

**Learning the memory policy with RL.**
[Memory-R1](https://arxiv.org/abs/2508.19828) (ACL 2026) trains the memory manager
itself: which of ADD / UPDATE / DELETE / NOOP to apply, and which memories to reason
over, with the only reward being whether the final answer was right. It needed just
**152 training QA pairs** to beat strong baselines and generalize across three
benchmarks and model scales from 3B to 14B, and a successor
([Memory-R2](https://arxiv.org/abs/2605.21768)) has already pushed past it. I read this
as the memory-layer version of the bitter lesson: hand-written heuristics for what to
remember are exactly the kind of thing that learning replaces.

**Governed evolving memory.** The [SSGM framework](https://arxiv.org/abs/2603.11768)
is a design proposal, not an implemented system, so cite it as an argument. But the
argument lands: decouple memory evolution from execution, admit only writes consistent
with a protected core of facts, decay by relevance under an explicit function, and
periodically reconcile the mutable store against an immutable ledger. Its formal claim
is that governed consolidation turns drift from compounding-per-step into
bounded-per-cycle.

**Value-aware retrieval** (RMM, [Memento](https://arxiv.org/abs/2508.16153), MemRL):
retrieval that learns from outcomes which memories actually helped.

**Temporal-causal structure** ([Zep's](https://arxiv.org/abs/2501.13956) validity
intervals, MAGMA's parallel temporal and causal graphs): the structural response to
failure mode 9.

**Verified self-evolving skills** (CoEvoSkills, SkillOS): procedural memory that
improves with use, with a verification gate before anything is reused, because an
unverified self-generated skill is just failure mode 1 with better packaging.

## Where this leaves us

Three positions I'll defend:

**The memory problem is real but mis-measured.** The gap between what vendors claim and
what anyone can independently verify is wider here than in any adjacent area I know.
Right now, the most useful contribution is probably not another memory system; it is
trustworthy measurement.

**The design principles have already converged**, even if no shipping system implements
them all: gate writes selectively; never let a summary be the only copy of the truth;
compress the index, not the evidence; treat every update as a decision with a cost;
learn retrieval from outcomes; give time a first-class representation. Read back through
the twelve failure modes and notice that each one violates at least one of these.

**The lifecycle is the open frontier.** Whether a memory gets better or worse with use
is the question that decides whether this product category delivers on its premise, and
until a few months ago nobody was measuring it at all.

I'm acting on the first position. I've started building an independent, pre-registered
benchmark of the commercial memory systems: adversarially validated judge, vendor-reviewed
configurations, full cost and latency accounting, methodology published *before* any
results. The harness is built and the judge validation is done. That's the next post.
