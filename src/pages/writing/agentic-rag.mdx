Naive RAG is the first thing everyone builds, and for good reason — it's three steps:
embed the question, pull the top few chunks from a vector store, paste them into the
prompt, and let the model answer. It works beautifully... right up until someone asks a
question that needs *two* lookups. Then it quietly falls apart.

**Agentic RAG** is what fills that gap. The idea is simple: instead of always retrieving
once and answering, you let the model decide *how* to retrieve — whether to retrieve at
all, what to look for, when to look again, and whether its own answer is actually backed by
what it found.

If naive RAG is a librarian who hands you the first book on the shelf, agentic RAG is a
researcher who checks a few sources, cross-references them, and tells you when they're not
sure.

## Where naive RAG falls over

The single-shot pipeline assumes the answer fits in one search against one source. In
practice, that assumption breaks all the time. The cases I keep running into:

- **Multi-hop questions.** "Which of the contracts signed in 2025 expires first?" needs
  several retrievals with reasoning in between — one search can't get you there.
- **Vague questions.** Short or underspecified queries embed poorly and pull back noise.
- **Scattered sources.** The answer might live in a database, a PDF, and a webpage — not all
  in one index.
- **Questions that need no retrieval at all.** "Thanks!" doesn't need a vector search, but a
  naive pipeline runs one anyway.
- **Confident but ungrounded answers.** The scariest one: a fluent answer the retrieved text
  doesn't actually support.

<Callout title="The failure that actually hurts">
In production, "no answer found" is annoying. A confident, well-written answer that *isn't
grounded in any source* is dangerous. Most of what follows exists to catch that one.
</Callout>

## So what makes it "agentic"?

Here's the mental shift. In naive RAG, the model is the last step of a fixed pipeline — it
generates, and that's it. In agentic RAG, the model is in the driver's seat: it takes
**actions** (retrieve, re-query, call a tool, answer), looks at what comes back, and decides
what to do next. It's the reason-and-act loop from ReAct — think a little, do a little,
repeat — instead of committing to one plan up front.

## The patterns worth knowing

You almost never need all of these. Pick the ones your questions actually demand.

### Routing — should we retrieve at all, and from where?

A lightweight router (a small classifier, or the model itself) looks at the query and picks
a path: answer directly, search vector store A or B, hit a SQL database, or go to the web.
The point is to match effort to difficulty — don't run an expensive search for "hello."

### Query transformation — fix the question before you search

Often the query you get isn't the query you should search with. So you reshape it first:
**rewrite** vague or conversational questions into search-friendly ones (resolving "it" and
"that"), **decompose** a multi-hop question into sub-questions you can tackle one at a time,
or use **HyDE** — generate a hypothetical answer and embed *that*, which usually retrieves
better than a terse question.

### Iterative retrieval — search, think, search again

For multi-hop questions, loop: answer a sub-question, use what you learned to ask the next
one, gather evidence, then synthesize. Each step sharpens the next search.

### Self-correction — grade, then retry

This is the pattern that actually buys you reliability. **Grade the retrieved docs** and
drop the irrelevant ones before generating (the core of CRAG). **Check the draft answer**
against its sources; if a claim isn't supported, retrieve again or revise (the idea behind
Self-RAG). And when nothing supports an answer, **say so** — "I don't have enough
information" beats a confident guess, especially in anything high-stakes.

### Tools beyond the vector store

Once retrieval is just another action, the retriever is one tool among many — web search, a
calculator, an internal API, a code interpreter. Agentic RAG is really tool-augmented
generation where one of the tools happens to be your knowledge base.

## What it looks like in code

Strip away the frameworks and it's a small state machine:

```python
def agentic_rag(question, max_steps=3):
    route = decide_route(question)          # answer directly? which source?
    if route == "answer":
        return generate(question)

    docs = []
    for _ in range(max_steps):
        hits = retrieve(question, source=route)
        docs = [d for d in hits if is_relevant(d, question)]   # grade
        if docs:
            break
        question = rewrite(question)         # nothing useful -> reformulate

    answer = generate(question, docs)
    if not grounded(answer, docs):           # check against the sources
        return "I don't have enough information to answer that."
    return answer
```

Two things make this *agentic* and not just a longer pipeline: the loop can retrieve again
based on what it sees, and the system is allowed to **stop early** — by answering directly,
or by refusing.

<Callout title="Bound the loop">
Every step is another model call: more latency, more cost, one more chance to wander off.
Cap the steps, cache aggressively, and run independent retrievals in parallel.
</Callout>

## It's not free

Before you reach for it, a reality check — "agentic" is not a synonym for "better":

- **Latency and cost** climb with every reasoning and retrieval step. A loop that *can* run
  five times can also *take* five times as long.
- **Evaluation gets harder.** Now you have to measure retrieval quality *and* answer
  faithfulness — ideally automatically, on every change — so a "smarter" prompt doesn't
  silently start dropping citations.
- **More parts, more failure modes.** Each router, grader, and rewrite step is one more
  thing to test and keep honest.

## When should you actually use it?

Reach for agentic RAG when questions are **multi-hop**, span **multiple sources**, are
**ambiguous**, or live in a **high-stakes domain** where citations and verification matter.
Stick with naive RAG for simple, single-fact lookups — it's faster, cheaper, and easier to
reason about.

My rule of thumb: start with the simplest pipeline that works, and add agency only where a
real failure forces you to. Routing and document grading give you the most reliability for
the least complexity. Full multi-step planning is worth it only when your questions
genuinely need it.
